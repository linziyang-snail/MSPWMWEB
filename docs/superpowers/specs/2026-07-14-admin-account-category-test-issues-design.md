# 人員權限設定後台測試問題前端改善設計

日期：2026-07-14

## 目標

處理業務單位提出的四項驗測問題，優先把前端可控制的資訊呈現、操作防呆與錯誤訊息做到完整，同時清楚界定必須由後端保證的資料規則。

本次涵蓋：

1. 人員管理新增「全部帳號」與「已駁回帳號」。
2. 改善已刪除帳號或科別重新新增時的前端行為與錯誤訊息。
3. 操作歷程同時呈現人員與科別管理紀錄。
4. 刪除科別前檢查科別下是否仍有人員帳號。

## 設計原則

- 前端不得用假成功或變更識別值的方式繞過後端限制。
- 涉及資料完整性的規則採保守策略：無法確認安全時不允許刪除。
- 清單、審核單與已刪除紀錄維持不同資料語意，不為了顯示方便混為同一種資料。
- 後端仍是唯一性、復原與刪除限制的最終權威；前端負責提早提示並正確呈現後端結果。

## 一、人員管理選單與清單

### 全部帳號

- 人員管理側邊選單新增「全部帳號」。
- 路由新增 `/accounts/all`，使用既有 `AccountStatusListView`。
- 此頁呼叫帳號清單 API 時不傳 `status`，直接顯示後端回傳的全部帳號資料。
- 前端不自行合併多個狀態查詢，也不自行排除後端回傳項目。
- 每筆資料依其實際狀態顯示「已啟用」、「需改密碼」、「已停用」等標籤。
- 待審核申請、已駁回申請及已刪除審核紀錄仍保留於各自頁面，不由前端額外併入全部帳號清單。

### 已駁回帳號

- 側邊選單新增「已駁回帳號」。
- 修正現有 `/accounts/rejected` 路由，不再導向 `/accounts/active`，改由 `AccountStatusListView` 顯示資料。
- 資料來源為 `GET /api/change-requests`，查詢條件至少包含：
  - `targetType=USER`
  - `status=REJECTED`
- 列表呈現申請日期、員編、姓名、科別、申請類型、申請人、駁回人及駁回原因；缺少的後端欄位以 `-` 呈現。
- 清單支援既有搜尋、排序與分頁模式。

## 二、已刪除帳號與科別重新新增

### 前端責任

- 新增表單不得因前端快取或歷史清單中存在相同員編、科別名稱而自行阻擋送出。
- 前端依正常新增流程送出資料，由後端判斷是否允許建立或恢復。
- 若後端允許建立，沿用既有成功流程與審核狀態更新。
- 若後端回傳已刪除資料造成的重複或唯一性衝突，將錯誤轉成可理解的訊息：
  - 帳號：「此員編曾被刪除，目前後端尚未開放重新建立，請聯絡系統管理人員恢復或解除限制。」
  - 科別：「此科別曾被刪除，目前後端尚未開放重新建立，請聯絡系統管理人員恢復或解除限制。」
- 若後端錯誤無法判定為上述情況，仍顯示後端提供的 `desc`，不誤導為重新建立限制。

### 後端必要修改

後端必須擇一提供可被驗證的正式行為：

1. 唯一性檢查只限制尚未刪除的帳號與科別，允許使用原員編或原科別名稱建立新資料；或
2. 提供恢復已刪除帳號、科別的 API，並保留原資料與操作歷程。

前端無法單獨修改資料庫唯一索引或保證重新新增成功。

## 三、操作歷程查詢

### 資料流

`operationHistoryService` 分別取得：

- `targetType=USER` 的人員管理異動。
- `targetType=ORGANIZATION` 的科別管理異動。

兩類查詢均包含 `CREATE`、`UPDATE`、`DELETE`，以及 `PENDING`、`APPROVED`、`REJECTED`、`CANCELED`。服務層抓取各自所有分頁後正規化資料，合併並依顯示日期由新到舊排序。

### 頁面行為

- 類型篩選提供「全部類型」、「人員帳號」與「科別」。
- 保留動作、狀態、關鍵字與日期區間篩選。
- 關鍵字可匹配操作者與被異動帳號或科別。
- 合併後再由前端進行顯示分頁。
- `rowKey` 必須包含 `targetType`，避免不同類型的 change request ID 相同時產生 Vue key 衝突。

### 部分失敗

- 使用 `Promise.allSettled` 或等價流程獨立處理兩類請求。
- 任一類成功時仍顯示成功取得的紀錄，並顯示「部分操作歷程載入失敗，請稍後重試」。
- 兩類都失敗時顯示載入失敗狀態，不把錯誤誤呈現為空資料。

## 四、科別刪除防呆

### 前端操作

- 使用者點擊科別刪除按鈕後，前端開啟既有確認視窗，不下載全部帳號自行判斷科別是否有人。
- 使用者確認後才呼叫刪除 API，送出期間以 loading 狀態防止重複操作。
- 是否允許刪除完全以後端回應為準，避免前端因分頁、快取或同時異動而誤判。

### 送出後錯誤處理

- 送出刪除申請時若後端回傳「科別下仍有人員」，顯示「此科別下仍有人員，請先刪除或移轉所有帳號。」
- 後端拒絕時保留選取科別與確認視窗，使使用者能看見錯誤原因；不得把拒絕誤呈現為完成或空資料。

### 後端必要修改

- `DELETE /api/organizations/{id}` 或核准科別刪除的流程，必須再次檢查科別下是否仍有帳號。
- 若有人員存在，後端應拒絕操作並回傳可辨識的錯誤代碼或明確 `desc`。
- 前端不以帳號清單 API 作為刪除資格判斷來源。

## 錯誤與載入狀態

- 全部帳號與已駁回帳號沿用全域 API 錯誤處理，空資料與載入失敗必須有不同畫面語意。
- 科別刪除送出期間禁用重複操作。
- 操作歷程支援部分成功提示，不因單一資料來源失敗而清空另一來源。
- 重新新增衝突只有在錯誤代碼或訊息可明確辨識時才套用專用文案。

## 測試與驗收

### 單元／元件測試

- 全部帳號路由不帶 `status` 查詢，且呈現後端回傳的各種帳號狀態。
- 已駁回帳號路由查詢 `USER + REJECTED`，不再重新導向。
- 已駁回清單正確呈現駁回原因與缺值占位符。
- 操作歷程同時查詢 `USER` 與 `ORGANIZATION`，合併後依日期排序。
- 操作歷程單一請求失敗時保留另一類資料並顯示部分失敗提示。
- 點擊科別刪除時不呼叫帳號清單 API，直接進入既有確認流程。
- 後端回傳科別下有人員時保留確認視窗並顯示明確提示。
- 後端允許時可正常完成刪除申請。
- 可辨識的已刪除資料衝突顯示專用訊息，其他錯誤保留後端訊息。

### 驗收情境

1. 管理員可從側邊選單進入「全部帳號」，看到後端回傳的全部現存帳號與正確狀態。
2. 管理員可進入「已駁回帳號」，確認是否有遭駁回的申請及其原因。
3. 操作歷程同時出現人員與科別的新增、修改、刪除紀錄。
4. 科別下仍有人員時，後端拒絕刪除，前端明確說明處理方式。
5. 後端確認科別可刪除時，前端正常完成刪除申請。
6. 使用已刪除的員編或科別名稱重新新增時，前端不預先誤擋；結果依後端能力正常成功或顯示明確限制訊息。

## 不在本次前端範圍

- 修改後端資料庫唯一索引。
- 實作帳號或科別復原 API。
- 保證科別刪除的交易一致性；此責任必須由後端處理。
- 將已刪除審核紀錄混入「全部帳號」主檔清單。
